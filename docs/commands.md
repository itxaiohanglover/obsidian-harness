# Persona — 命令详解

> 版本：v0.5.0 | 更新时间：2026-05-19

---

## 设计理念

Persona 的命令体系基于一个洞察：**用户不应该记住命令名，而应该感知自己的状态。**

| 你的状态 | 命令 | 隐喻 |
|---------|------|------|
| 脑子里一团糟，不知道从哪开始 | `/um` | "嗯…"（困惑的声音） |
| 想看看进度，理清方向 | `/aha` | "啊哈！"（顿悟的声音） |
| 要换个工作模式 | `/go` | 走向新场景 |
| 积累够多了，想沉淀 | `/distill` | 蒸馏精华 |
| 想建一个新的工作模式 | `/new` | 从零创造 |

命名原则：
- 足够短（2-3 个字符），好打字
- 有情绪感，不是功能描述
- 符合直觉，不需要查文档

## /um — 发散

### 一句话

外化混乱，发现方向。

### 设计理念

`/um` 对应双钻石的发散阶段。用户的混乱不是问题——混乱是原材料。AI 的角色是帮你把脑子里的东西倒出来，发现其中的结构和方向，但**不替你做决定**。

核心原则：
- **诊断先于行动** — 先一句话说发现了什么，再问要不要动手
- **不过度组织** — 保留用户原始表达的声音
- **等待选择** — 给方向不给答案，等用户 pick

### 使用方式

#### 无输入：`/um`

AI 会做两件事：

1. **变更检测**（轻量）— 自上次 `/go` 切换以来，vault 中哪些文件变了
2. **方向发现**（如果无变更）— 扫描 vault 找孤儿笔记、未标签内容、过期笔记等

输出示例：
```
自上次切换后，你改了 3 个文件：[[设计方案]]、[[会议纪要-0519]]、[[TODO]]
要从这些变更继续，还是探索新方向？
```

#### 有输入：`/um <target>`

| 输入 | AI 行为 |
|------|---------|
| `看看改了什么` | 变更检测（即使有输入也优先） |
| `[[笔记名]]` 或文件路径 | **Focus/Dive 模式** — 渐进深入 |
| 项目名 | 分析项目笔记缺口，建议重构 |
| 时间范围（"最近一周"） | 收集该时段内容，发现主题 |
| 主题名 | 新探索 — 构建框架，列核心概念 |
| 大段文本（>500字） | Brain dump — 拆主题，创建链接笔记 |

#### Focus/Dive 模式

当输入是 wikilink 或已有笔记名时，进入渐进深入：

```
📄 架构设计
├── 模块划分 (已有子笔记 [[模块划分]])
├── 数据流 (建议拆分 → 可创建 [[数据流]])
└── 部署方案 (内容较少，保留)

要深入哪个主题？
```

用户选择后：
- 创建子笔记，加 backlink
- 父笔记中替换为链接 + 一句话摘要
- 询问：继续深入还是回到父笔记？

**这实现了你的核心工作模式：宏观想法 → 拆子文件 → 链接 → 聚焦 → 继续深入**

### _actions 路由

如果用户输入匹配当前场景 `_actions` 中的 trigger 关键词（大小写不敏感），则执行对应 action 的 prompt，而非标准 /um 行为。

示例：在 research 场景下输入 `/um 对比` → 匹配 `compare` action → 执行对比分析流程。

## /aha — 收敛

### 一句话

汇总进度，提炼清晰，建议下一步。

### 设计理念

`/aha` 对应双钻石的收敛阶段。用户已经有了素材，需要的是：一句话看到全貌，一个建议知道下一步。

核心原则：
- **一次扫描，一次回复** — 快速，不啰嗦
- **像接续对话** — 简短，用用户的语言
- **如果用户在做事中途** — 续做而非只总结

### 使用方式

#### 无输入：`/aha`

个人站会模式：

1. 扫描自上次 `/go` 切换以来的 vault 变更
2. 一句话总结："自切换场景以来，你改了 N 个文件，完成了 N 个任务，项目X还差Y部分"
3. 建议一个下一步行动
4. 更新 `last_aha_at` 时间戳（让 /um 知道上次收敛时间）

#### 有输入：`/aha <scope>`

| 输入 | AI 行为 |
|------|---------|
| 时间范围（"最近一周"） | 总结该时段产出，生成 review |
| 项目名 | 项目状态报告：done / pending / at risk |
| "今天的进展" | 检查最近修改的调研笔记，提取未完成任务 |
| 其他文本 | 总结 vault 中与该主题相关的一切 |

### 蒸馏检查

每次 /aha 执行完毕后，如果本次对话中 AI 被用户明确纠正过：

- 提示："要记住这个偏好吗？"
- 用户确认 → 追加到 `_memory`
- 用户说"记住..." → 立即追加

## /go — 切换场景

### 一句话

切换工作人格，进入新的思考模式。

### 设计理念

场景切换不只是加载不同 prompt——它是一种**心理转换**。从"我是工程师"变成"我是策展人"。/go 的 on-enter hook 帮助用户快速进入状态。

核心原则：
- **一行确认** — 切换要快，不要冗长输出
- **依赖懒检查** — 缺了才说，不预先列所有
- **继承透明** — 如果场景有继承，明确告知来源

### 使用方式

#### 切换场景：`/go <scene>`

```
/go coding
→ ✓ 已切换到 coding。
   可用动作：project-note（创建项目笔记）、dev-log（开发日志）、architecture（架构图）...
```

#### 列出可用场景：`/go`（无参数）

```
可用场景：
  research       — 调研探索：信息收集、对比分析、综合报告
  coding         — 编码文档
  patent-writing — 专利撰写
  my-research    — [自建] 文献调研
```

#### 调试模式：`/go <scene> --dry-run`

展示场景的完整 prompt 组合（继承链、合并后的 _um/_aha、_actions 等），不实际切换。用于调试继承链。

### 首次使用

用户第一次在 vault 中使用 `/go` 时：

1. 自动创建 `.persona/` 目录结构
2. 创建 `profile.md` 模板（引导用户写工作画像）
3. 友好提示下一步

### 模糊匹配

支持模糊场景名：
- `code` → `coding`（前缀匹配）
- `patent` → `patent-writing`（前缀匹配）
- `resarch` → `research`（编辑距离 ≤ 2）

## /distill — 画像精炼

### 一句话

把零散记忆蒸馏成结构化画像。

### 设计理念

`_memory` 是日积月累的原始偏好记录（"我喜欢简短回复"、"不要用表情"、"代码注释用英文"）。时间长了会冗余、矛盾、过时。`/distill` 把这些原材料精炼成精准的 `_profile`。

核心原则：
- **必须展示 diff** — 用户看到变了什么才确认
- **不删用户没同意删的** — 每条记忆的去留由用户决定
- **保留用户的声音** — 不要把口语偏好改写成正式规则

### 使用方式

```
/distill
```

流程：
1. 读取 `_memory`（需要 ≥ 3 条才启动）
2. 分析模式：重复 3+ 次的提升为 profile、矛盾的请用户裁决、一次性的保留
3. 生成新 `_profile` 草稿
4. 展示 diff（旧 vs 新 + 变更说明）
5. 等用户确认 / 修改 / 取消

### 什么会被提升

| 记忆模式 | 处理方式 |
|---------|---------|
| "我喜欢X" 出现 3+ 次变体 | 提升为 _profile 中的明确偏好 |
| 同一种纠正重复多次 | 提升为规则 |
| 一次性的具体指令 | 保留在 _memory，不提升 |
| 与现有 _profile 矛盾 | 请用户裁决 |

## /new — 创建场景

### 一句话

引导式创建你的专属工作人格包。

### 设计理念

创建场景不应该需要手写 JSON。`/new` 通过 7 个问题（逐一交互，不一次性倾倒）引导用户完成场景创建。

如果用户选择继承，跳过已继承的部分只问差异。

### 使用方式

```
/new
```

交互流程：
1. **场景名称** — 英文短横线命名
2. **一句话描述** — 场景做什么
3. **是否继承** — 基于已有场景扩展？
4. **需要哪些 Skills** — 从 registry.json 列出可选项
5. **使用习惯** — 描述典型工作流程（→ 初始 `_profile`）
6. **每个 Skill 怎么用** — 特殊风格要求（→ skill overlays）
7. **专属动作** — 快捷操作（→ `_actions`）

### 输出

自动生成到 `{vault}/.persona/scenes/{name}/`：
- `manifest.json`
- `prompts.json`（含自动生成的 `_um` 和 `_aha`）
- `{vault}/.persona/contexts/{name}.md`（上下文模板）

### 快速继承

如果选择继承已有场景，Q4-Q6 中已继承的部分自动跳过，只问差异。

## 命令间协作

```
/go research       ← 进入场景，加载人格
    ↓
/um                ← 发散：发现方向、外化混乱
    ↓
(用户在 Obsidian 中工作)
    ↓
/aha               ← 收敛：汇总进度、建议下一步
    ↓                  └── 蒸馏检查：发现纠正 → 存 _memory
(积累 N 次)
    ↓
/distill           ← 精炼：_memory → _profile
    ↓
(下次 /um 更懂你)
```

场景间流转：
- 在 research 中讨论代码架构 → AI 建议 `/go coding`
- 在 coding 中想做调研对比 → AI 建议 `/go research`
- 只建议不强制，用户忽略则不再重复
